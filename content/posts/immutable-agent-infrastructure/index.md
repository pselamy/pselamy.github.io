A trading agent should not be able to edit its own risk limits.

That sounds obvious. It was obvious before it happened, too.

The agent had a simple daily loss cap. Stop trading if the day goes worse than a fixed amount. The cap lived in a file inside the agent workspace because that was convenient during development. The agent could read it. The process could load it. The file lived next to the scripts it was allowed to modify.

Then the agent changed it.

Not maliciously. Not dramatically. It was working through a task, touched the wrong config-adjacent file, and rewrote a loss limit from `$-50` to `$-5`. In this specific case the mistake made the system more conservative. That was lucky. The same class of bug could have moved the number the other way.

That was the moment I stopped treating mutable agent filesystems as a developer convenience and started treating them as a production risk.

## the bug was architectural

The first instinct is to patch the agent.

Tell it not to edit risk files. Add a prompt rule. Add a check. Rename the file. Put a comment at the top that says do not touch this.

Those are all weak fixes.

If an agent can write to the file that defines its authority, then the boundary is fake. You do not have a risk limit. You have a suggestion stored on disk.

The bug was not that the agent picked the wrong file. The bug was that the filesystem allowed the wrong file to be picked.

## mutable workspaces are seductive

Mutable workspaces feel great early on.

You can ssh into the machine, edit a script, restart the process, and keep moving. Agents can patch themselves. Config can sit next to code. Logs can land wherever. Everything is fast because everything is informal.

That informality is exactly what makes it dangerous.

Once an agent can run tools, write files, recover from errors, and keep working without you watching every step, the workspace becomes part of the permission model. It is not just storage. It is authority.

A writeable file might be code. It might be memory. It might be credentials. It might be a loss cap. If the system does not make those categories physically different, the agent will eventually blur them.

## prompts are not permission boundaries

I still use prompt rules. They are useful.

But prompt rules are not access control.

A prompt can say "do not edit this file." A mount can make the file read-only. Those are not equivalent.

The model can misunderstand the prompt. A tool can apply a broad patch. A retry can lose context. A refactor can move files around. A bad instruction from me can point the agent at the wrong path.

The operating system is less polite. If the path is read-only, the write fails.

That is the kind of failure I want.

## the fix was GitOps plus overlays

The durable fix was to move the agents onto a Kubernetes setup managed through GitOps.

Each agent gets a small chart. The desired state lives in git. The cluster reconciles to that state. Runtime data goes on persistent volumes. Code and config come from versioned sources. Deployments are reviewable. Rollbacks are normal.

The important detail was the file layout.

Agent scripts are mounted into the workspace as immutable overlays using Kubernetes `subPath` mounts. The agent still sees the files at the paths it expects. The process can still execute them. But the mounted script files come from controlled sources and are not just loose files on the writable volume.

The writable volume is still there, but it is for state: logs, task records, caches, ledgers, and artifacts.

The distinction matters:

- code comes from git
- risk limits come from deployment config
- runtime state goes on the volume
- agent edits happen only where edits are allowed

That is boring infrastructure. It is also the difference between "please do not touch that" and "you cannot touch that from here."

## why subPath mattered

The awkward part of migrating agent systems is that they often expect a normal filesystem.

A script expects to live at `/home/agent/bin/run`. A tool expects a config file at a specific path. A long-running process expects its working directory to be writable. If you make the whole directory read-only, half the system breaks. If you leave the whole directory writable, the boundary is gone.

`subPath` overlays gave me a practical middle ground.

I could mount specific files into exact paths inside an otherwise writable workspace. The agent did not need a large rewrite. The path contract stayed the same. The mutability contract changed underneath it.

That is the kind of migration I like: change the safety property without asking every caller to learn a new layout.

## risk limits belong outside the agent

A trading agent is a clean example because the stakes are easy to understand.

The agent can make decisions inside a box. It can evaluate markets, place orders, reconcile fills, and stop when the rules say stop. But the rules that define the box should not be authored by the same process operating inside it.

My current pattern is simple:

- hard caps live in deployment config
- the process receives them as environment variables or mounted read-only config
- the agent can report on them, but not rewrite them
- changing them requires a git change and deployment reconciliation

That does not make the trading system safe by itself. Nothing does.

It does remove one stupid failure mode: the worker cannot casually redefine its own limits while doing unrelated work.

## this applies beyond trading

Trading just made the lesson obvious.

The same pattern applies to any agent with real authority.

A coding agent should not be able to rewrite the policy that says which repos it can push to. A career agent should not be able to send external messages just because it drafted one. A finance monitor should not be able to change the threshold that decides whether something is alarming.

The more useful an agent becomes, the more tempting it is to give it a big writable workspace and let it figure things out.

I think that is backwards.

Useful agents need smaller boxes, not bigger ones. More context, yes. Better tools, yes. But authority should be carved into hard surfaces.

## what I would do earlier next time

If I were starting the system again, I would separate these categories from day one:

1. immutable code
2. immutable policy
3. mutable state
4. generated artifacts
5. human approval points

I would also make the default path boring:

- every agent deployed from git
- every risky setting defined outside the workspace
- every writable mount treated as untrusted state
- every production change visible in a diff
- every rollback one commit away

That sounds slower than hacking on a box.

It is slower for the first afternoon. It is faster the first time an agent does something surprising and the blast radius stops at the boundary you already built.

## the larger lesson

Agents make bad infrastructure choices visible.

A human developer can know not to edit the wrong file. A human can notice that a config value looks dangerous. A human can hold a bunch of informal rules in their head and mostly obey them.

Agents are worse at that. They need the system to express the rules directly.

That has changed how I think about agent infrastructure. I care less about whether an agent can recover from a mistake by reasoning harder. I care more about whether the mistake is possible in the first place.

The fix for the trading agent was not a smarter prompt.

It was an immutable boundary.
